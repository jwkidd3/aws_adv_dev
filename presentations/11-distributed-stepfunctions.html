Module 11

Distributed Complexity & Step Functions

Orchestrating the Cloud Air booking flow without distributed spaghetti.

Why Distributed Systems Are Hard

  • Partial failure — any node can fail independently; the system must cope
  • Network partitions — messages arrive out of order, duplicated, or not at all
  • Distributed transactions — two-phase commit doesn't scale in microservices
  • Observability gaps — a chain of Lambda invocations has no built-in audit trail
  • State sprawl — each service tracking "where are we in the workflow?" leads to drift

The Cloud Air booking flow (search → hold → pay → confirm → notify) spans 5 services — a canonical distributed-complexity problem.

The Saga Pattern

  • Replaces a distributed ACID transaction with a sequence of local transactions, each publishing an event
  • On failure: compensating transactions undo earlier steps (release held seat, void payment)
  • Two variants: Choreography (event-driven, each service reacts) vs Orchestration (central coordinator drives the flow)

Choreography scales well but debug complexity grows with step count. Orchestration trades some autonomy for an explicit, inspectable execution record — the sweet spot for booking workflows.

AWS Step Functions

  • Fully managed orchestration service — state machines defined in Amazon States Language (ASL)
  • Standard Workflows — exactly-once, auditable, up to 1 year; suitable for bookings
  • Express Workflows — at-least-once, high throughput, up to 5 min; suitable for streaming ingestion
  • Native integrations: Lambda, DynamoDB, SQS, SNS, ECS, SageMaker, EventBridge — no glue code
  • Visual debugger in the console shows exactly which state failed and its I/O

ASL: Amazon States Language

{
  "Comment": "CloudAir booking saga",
  "StartAt": "HoldSeat",
  "States": {
    "HoldSeat": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:ACCT:function:cloudair-$USER_ID-hold",
      "Next": "ProcessPayment",
      "Catch": [{ "ErrorEquals": ["States.ALL"], "Next": "ReleaseSeat" }]
    },
    "ProcessPayment": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:ACCT:function:cloudair-$USER_ID-pay",
      "Next": "ConfirmBooking",
      "Retry": [{ "ErrorEquals": ["PaymentTimeout"], "MaxAttempts": 2, "IntervalSeconds": 3 }]
    },
    "ConfirmBooking": { "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:ACCT:function:cloudair-$USER_ID-confirm",
      "End": true }
  }
}

Catch enables compensating transactions; Retry handles transient failures before giving up.

State Types at a Glance

  • Task — invoke a Lambda, SDK integration, or HTTP endpoint
  • Choice — conditional branching on input JSON values
  • Parallel — fan-out concurrent branches; merge all outputs
  • Map — iterate over an array of items, running the same branch per item
  • Wait — pause until a timestamp or duration (no Lambda idling)
  • Pass / Succeed / Fail — flow control and terminal states

Map + Parallel replace hand-rolled asyncio/concurrent.futures fan-out patterns.

SDK Integrations — No Lambda Glue Needed

"NotifyPassenger": {
  "Type": "Task",
  "Resource": "arn:aws:states:::sns:publish",
  "Parameters": {
    "TopicArn": "arn:aws:sns:us-east-1:ACCT:cloudair-$USER_ID-notify",
    "Message.$": "States.Format('Booking {} confirmed', $.bookingId)"
  },
  "Next": "Done"
}

Optimistic SDK integrations (the arn:aws:states:::service:action form) call the AWS API directly — no Lambda invocation cost, no cold starts.

Orchestration vs Choreography

  • Orchestration — Step Functions holds state; audit trail is free; easy to debug and replay
  • Choreography — each service reacts to events; loose coupling; harder to trace across services
  • Hybrid is common: Step Functions orchestrates within a bounded context; EventBridge choreographs between contexts
  • Cloud Air uses both: Step Functions owns the booking saga; EventBridge fires domain events (booking.confirmed) to downstream systems

Starting & Monitoring Executions

# Start an execution
aws stepfunctions start-execution \
  --state-machine-arn arn:aws:states:us-east-1:$ACCT:stateMachine:cloudair-$USER_ID-booking \
  --input '{"flightId":"CA-404","passengerId":"u42","seats":1}'

# List running executions
aws stepfunctions list-executions \
  --state-machine-arn arn:aws:states:us-east-1:$ACCT:stateMachine:cloudair-$USER_ID-booking \
  --status-filter RUNNING

# Describe execution history (audit trail)
aws stepfunctions get-execution-history \
  --execution-arn arn:aws:states:us-east-1:$ACCT:execution:cloudair-$USER_ID-booking:exec-1

Step Functions + SAM

BookingStateMachine:
  Type: AWS::Serverless::StateMachine
  Properties:
    DefinitionUri: statemachine/booking.asl.json
    DefinitionSubstitutions:
      HoldFunctionArn: !GetAtt HoldSeatFunction.Arn
      PayFunctionArn:  !GetAtt ProcessPaymentFunction.Arn
    Policies:
      - LambdaInvokePolicy:
          FunctionName: !Ref HoldSeatFunction
      - LambdaInvokePolicy:
          FunctionName: !Ref ProcessPaymentFunction
    Logging:
      Destinations: [{ CloudWatchLogsLogGroup: { LogGroupArn: !GetAtt SFNLogGroup.Arn } }]
      Level: ALL

SAM substitutions inject Lambda ARNs into the ASL at deploy time — keep ASL portable across environments.
