Module 2

The Cloud Journey: Monolith & Migration

Off-cloud architecture, the Cloud Air monolith, the 6 Rs, and guardrails for the migration ahead.

Cloud Air — The Starting Point

  • A single Python/Flask application running on a bare-metal server in a co-lo
  • One PostgreSQL database — shared by every feature: bookings, inventory, loyalty, payments
  • Deployments require: schedule a maintenance window, quiesce traffic, rsync code, restart gunicorn
  • Scaling means calling the data-center team to provision a new box — minimum 2-week lead time

Any of this sound familiar? This is the architecture this course is designed to retire.

Monolith Pain Points

Tight coupling
A change in the loyalty module requires a full-stack regression test for bookings
Scaling bottleneck
Seasonal peak (holidays) demands 10× capacity for the search service; you scale everything
Long release cycles
Feature flags are a spreadsheet. Rollbacks mean reverting the whole app
Single point of failure
DB connection pool exhaustion during a booking surge takes down the loyalty portal too

Off-Cloud to Cloud: The Migration Decision

  • Trigger: co-location contract renewal + aggressive SLA commitments Cloud Air can't meet on current infrastructure
  • Goal: elastic scale, independent deployability, observability — without a "big bang" rewrite
  • The migration will be incremental — the monolith stays in service while individual capabilities are extracted

This is not a lift-and-shift exercise. The architecture will change. The application contract (REST API) stays stable for external consumers throughout.

The 6 Rs of Migration

Retire
Decommission what's no longer needed
Retain
Keep on-premises for now (compliance, latency)
Rehost
Lift-and-shift to EC2 — fast, low risk, low reward
Replatform
Lift-tinker-shift, e.g. move to RDS without code changes
Repurchase
Move to SaaS (e.g. replace CRM with Salesforce)
Refactor / Re-architect
Decompose into microservices — highest cost, highest ROI

Cloud Air's booking engine: Refactor. Its legacy reporting tool: Retire. Its compliance archive: Retain on-premises for now.

Cloud Air Migration Phases

  • Phase 0 — Environment & IaC baseline (Lab 1 & 2): Cloud9, CLI/SDK smoke test, CloudFormation stack, Elastic Beanstalk monolith deploy
  • Phase 1 — Twelve-Factor compliance (Lab 3): Externalise config to SSM; migrate secrets to Secrets Manager
  • Phase 2 — First microservice extraction (Lab 4): Lambda + API Gateway flight-search service; Strangler Fig proxy routes around the monolith
  • Phase 3 — Persistence & orchestration (Labs 5–6): DynamoDB replaces the shared DB; Step Functions; event-driven messaging
  • Phase 4 — Secure & observe (Lab 7): Cognito JWT, X-Ray tracing, full cleanup

AWS Migration Guardrails

  • IAM permission boundary — student roles scoped to us-east-1 via RestrictToUsEast1 policy; prevents runaway resource creation
  • Resource tagging — all resources require Owner: cloudair-userN tag; enables cost attribution and automated cleanup
  • Service Control Policies (SCPs) — org-level guardrails block certain high-cost or out-of-scope services entirely
  • AWS Config rules — continuous compliance checks flag untagged or public-access resources immediately

Guardrails are not bureaucracy — they are the contractual boundary that lets you move fast without wrecking a shared environment.

The Cloud Air Target Architecture

  • API Gateway — single external entry point; routes per path prefix to microservices or the legacy proxy
  • Lambda functions — stateless, independently deployed; one per bounded context (search, booking, loyalty, payment)
  • DynamoDB — single-table design per service; no shared DB
  • SQS / SNS / EventBridge — async event bus replaces synchronous inter-module calls
  • Step Functions — orchestrates multi-step booking saga with rollback

The monolith is the starting point, not the destination. Each Lab advances the decomposition.

Cloud9 manages temporary credentials automatically. No aws configure needed. If get-caller-identity returns your role ARN, you're ready.
