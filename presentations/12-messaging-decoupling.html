Module 12

Decentralized Data & Messaging

SQS, SNS, Kinesis, IoT, and EventBridge — decoupling Cloud Air's services so failures don't cascade.

Decentralized Data in Microservices

  • Each microservice owns its data store — no shared database, no shared schema
  • Cross-service queries become async messaging or API calls — never direct DB joins
  • Data consistency shifts from ACID transactions to eventual consistency via events
  • The Cloud Air monolith had one PostgreSQL instance; the microservices split it across DynamoDB (bookings), Aurora Serverless (billing), S3 (manifests)

The right question is not "which DB?" but "which service owns this data?" — ownership determines messaging topology.

Amazon SQS — Reliable Queuing

  • Standard queues — nearly unlimited throughput, at-least-once delivery, best-effort ordering
  • FIFO queues — exactly-once processing, strict ordering within message group
  • Visibility timeout decouples receive from delete — consumer has time to process before the message reappears
  • Dead-letter queues (DLQ) capture messages that exceed maxReceiveCount — prevents poison-pill loops
  • Lambda event source mapping polls SQS automatically — no polling code needed

SQS is the right choice when one consumer should process each message (point-to-point). For fan-out, pair with SNS.

SQS with boto3

import boto3, os, json

sqs = boto3.client("sqs", region_name="us-east-1")
Q_URL = os.environ["BOOKING_QUEUE_URL"]

# Producer — enqueue a booking request
sqs.send_message(
    QueueUrl=Q_URL,
    MessageBody=json.dumps({"bookingId": "BK-007", "seats": 2}),
    MessageAttributes={
        "eventType": {"StringValue": "BookingRequested", "DataType": "String"}
    }
)

# Consumer — receive and delete
msgs = sqs.receive_message(QueueUrl=Q_URL, MaxNumberOfMessages=5,
                            WaitTimeSeconds=20)["Messages"]
for m in msgs:
    process(json.loads(m["Body"]))
    sqs.delete_message(QueueUrl=Q_URL, ReceiptHandle=m["ReceiptHandle"])

Long-polling (WaitTimeSeconds=20) reduces empty receives and API cost. Always delete after successful processing.

Amazon SNS — Pub/Sub Fan-out

  • Topics deliver one message to N subscribers simultaneously: SQS queues, Lambda, HTTP/S, email, SMS, mobile push
  • Message filtering — subscribers declare a JSON filter policy; SNS evaluates before delivery — saves Lambda invocations
  • SNS → SQS fan-out — canonical pattern: one SNS topic fans out to per-consumer SQS queues, giving each service an independent, buffered feed
  • FIFO topics + FIFO queues for ordered fan-out with exactly-once semantics
sns = boto3.client("sns", region_name="us-east-1")
sns.publish(
    TopicArn=os.environ["BOOKING_TOPIC_ARN"],
    Message=json.dumps({"bookingId": "BK-007", "status": "CONFIRMED"}),
    MessageAttributes={"eventType": {"DataType": "String",
                                      "StringValue": "booking.confirmed"}}
)

Amazon Kinesis Data Streams

  • Ordered, partitioned stream — records are retained up to 365 days (replay at any time)
  • Shard — unit of capacity: 1 MB/s in, 2 MB/s out per consumer; scale by resharding
  • Partition key determines shard placement — use passengerId to keep per-passenger events in order
  • Multiple enhanced fan-out consumers each get 2 MB/s — no throughput sharing
  • Lambda event source: checkpointed automatically; failed batch goes to DLQ or bisection mode

Kinesis vs SQS: choose Kinesis when you need replay, ordered per-entity processing, or multiple independent consumers reading the same stream at their own pace.

AWS IoT Message Broker

  • MQTT/HTTPS broker for device-to-cloud and cloud-to-device messaging at scale
  • Rules Engine — SQL-like rules route messages to Lambda, Kinesis, DynamoDB, S3, SNS
  • Cloud Air cabin IoT: seat sensors publish telemetry to cloudair/cabin/+/seat; rules route anomalies to the maintenance Lambda
  • Thing Registry + X.509 certificates + IoT policies enforce per-device authorization
{
  "sql": "SELECT *, topic(3) AS seatId FROM 'cloudair/cabin/+/seat' WHERE temp > 35",
  "actions": [{
    "lambda": { "functionArn": "arn:aws:lambda:us-east-1:ACCT:function:cloudair-seat-alert" }
  }]
}

Amazon EventBridge

  • Serverless event bus — producers emit events; rules route to targets (Lambda, SQS, Step Functions, API destinations)
  • Event pattern matching — rules filter on any field in the event JSON, reducing noisy Lambda invocations
  • Schema Registry — auto-discover and version event schemas; generate SDK bindings
  • Archive & replay — store all events; replay to test new consumers against historical data
  • Cross-account / cross-region — event buses connect separate AWS accounts cleanly

EventBridge is the preferred choreography spine for Cloud Air's inter-service domain events — decoupled by design, no consumer list in the producer.

Publishing to EventBridge

events = boto3.client("events", region_name="us-east-1")

events.put_events(Entries=[{
    "Source":       "cloudair.booking",
    "DetailType":   "booking.confirmed",
    "Detail": json.dumps({
        "bookingId":   "BK-007",
        "passengerId": "u42",
        "flightId":    "CA-404",
        "seats":       2
    }),
    "EventBusName": f"cloudair-{os.environ['USER_ID']}-bus"
}])

Producers know nothing about consumers. Adding a new downstream service means adding a new EventBridge rule — zero producer changes.

Choosing the Right Service

  • SQS — point-to-point queuing, buffering, load leveling, DLQ for poison pills
  • SNS — push fan-out to multiple heterogeneous endpoints simultaneously
  • Kinesis — ordered streams, replay, multiple independent consumers, high-throughput analytics
  • IoT Core — device connectivity, MQTT, rules-based routing from edge to cloud
  • EventBridge — domain events between services, schema registry, cross-account routing, SaaS integrations

These are not mutually exclusive — SNS → SQS fan-out into EventBridge rules is a valid and common pattern for Cloud Air's notification pipeline.
