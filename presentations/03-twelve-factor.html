Module 3

The Twelve-Factor App & Architectural Patterns

The methodology that underpins every refactoring decision in the Cloud Air migration.

Why the Twelve-Factor App?

  • Published by Heroku engineers in 2011 — distills patterns common to SaaS apps that survive at scale
  • Not prescriptive about language or framework — applicable to Python, Java, Go, Node equally
  • Cloud Air violates at least 7 of the 12 factors in its current form — we'll map each one
  • Compliance is not optional for serverless: Lambda enforces several factors by design (stateless, ephemeral, env-driven)

Factors I–IV: Codebase, Dependencies, Config, Backing Services

I — Codebase
One repo, many deploys. Cloud Air has prod, staging and a "demo" branch deployed from laptops. ❌ Fix: single main branch; deploy artefacts only.
II — Dependencies
Explicitly declare & isolate. Cloud Air relies on a globally installed psycopg2. ❌ Fix: requirements.txt + venv.
III — Config
Store config in the environment, never in code. DB password in config.py. ❌ Fix: SSM Parameter Store + env vars at deploy time.
IV — Backing Services
Treat DB, cache, queue as attached resources accessed by URL/credentials. ✅ Already loosely coupled, but DB is hardcoded to localhost.

Factors V–VIII: Build, Process, Port, Concurrency

V — Build, Release, Run
Strictly separate stages. Cloud Air SSH-deploys .py files directly to prod. ❌ Fix: CodeBuild artefact → CodeDeploy run stage.
VI — Processes
Stateless, share-nothing. Cloud Air stores session data in a module-level dict. ❌ Fix: ElastiCache for sessions; Lambda is stateless by design.
VII — Port Binding
Export services via port binding, not container injection. Flask dev server is used in prod behind Nginx. ❌ Fix: Gunicorn on Beanstalk; Lambda HTTP via API GW.
VIII — Concurrency
Scale out via the process model. Cloud Air scales by adding threads. ❌ Fix: Lambda scales by adding process instances automatically.

Factors IX–XII: Disposability, Dev/Prod Parity, Logs, Admin

IX — Disposability
Fast startup, graceful shutdown. Cloud Air's startup takes 40 s and holds DB connections during shutdown. Fix: decouple connection management; Lambda cold start optimisation.
X — Dev/Prod Parity
Keep dev, staging, prod as similar as possible. Local SQLite vs prod Postgres causes silent data-type bugs. Fix: RDS locally via Docker Compose or SAM local.
XI — Logs
Treat logs as event streams. Cloud Air writes to rotating files on disk. Fix: structured JSON to stdout → CloudWatch Logs via Lambda/Beanstalk.
XII — Admin Processes
Run admin/management tasks as one-off processes. DB migrations run inside the app on startup. Fix: migration task in a separate Lambda invocation or ECS task.

Cloud Air's compliance score before: 2/12. Target after Labs 1–3: 10/12.

Architectural Styles Relevant to Cloud Air

  • Monolith — single deployable unit; low operational complexity, high coupling; Cloud Air Day 0
  • Modular Monolith — enforced module boundaries within one process; stepping stone, not the destination
  • Microservices — independently deployable services with bounded contexts; Cloud Air target; higher operational complexity offset by independent scale and release
  • Serverless — functions-as-a-service + managed event bus; removes server management entirely; what Lambda + EventBridge give us

Key Architectural Patterns

Strangler Fig
Incrementally replace monolith functionality by routing traffic to new services
Saga
Manage distributed transactions with compensating actions — used in the booking flow via Step Functions
CQRS
Separate read and write models to optimise each path independently
Event Sourcing
Persist state changes as an immutable event log; rebuild state by replaying
Anti-Corruption Layer
Adapter between new microservices and legacy models — prevents old schemas from leaking in
API Gateway
Single entry point; centralised auth, rate limiting, routing, contract versioning

Config Externalisation — The Twelve-Factor Way

# BAD — Factor III violation: config in code
DB_HOST = "db.internal.cloudair.com"
DB_PASS = "sup3rS3cret"

# GOOD — read from environment at runtime
import os, boto3

DB_HOST = os.environ["DB_HOST"]          # injected by Beanstalk env vars

# Secrets: never in env vars either — use Secrets Manager
secret = boto3.client("secretsmanager") \
    .get_secret_value(SecretId="cloudair/prod/db")["SecretString"]

SSM Parameter Store (non-secret config) and Secrets Manager (credentials, API keys) are the AWS implementations of Factor III and Factor IV. Lab 3 wires both into Cloud Air.

Structured Logging — Factor XI in Practice

import json, logging, os

logger = logging.getLogger()
logger.setLevel(logging.INFO)

def handler(event, context):
    logger.info(json.dumps({
        "service": "flight-search",
        "user_id": os.environ.get("USER_ID"),
        "correlation_id": event.get("requestContext", {}).get("requestId"),
        "origin": event["queryStringParameters"]["origin"],
        "dest":   event["queryStringParameters"]["destination"],
    }))
    # ... search logic

Lambda automatically forwards stdout to CloudWatch Logs. JSON structure enables metric filters and Contributor Insights without any log agent.

Twelve-Factor Compliance Checklist for Cloud Air

  • ✅ I — Single repo (aws_adv_dev) after Lab 1a
  • ✅ II — requirements.txt pinned; Lambda layers for shared deps
  • ✅ III — No config in code after Lab 3a (SSM) and Lab 3b (Secrets Manager)
  • ✅ VI — Lambda is stateless; session state in ElastiCache/DynamoDB
  • ✅ VIII — Lambda scales horizontally by default; no thread pool tuning
  • ✅ XI — Structured JSON logs to CloudWatch Logs; X-Ray correlation in Lab 7b
  • ⚠️ X — Dev/prod parity remains aspirational until SAM local is configured in Lab 4a
