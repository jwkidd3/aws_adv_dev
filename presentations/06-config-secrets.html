Module 6

Application Configuration & Secrets Management

Twelve-Factor Factor III & IV in practice — externalising config and protecting credentials in the Cloud Air stack.

The Config Anti-Pattern

  • Cloud Air monolith: db_password = "Fl1ght$2024" hard-coded in app.config
  • Config in source control = credentials in every developer's laptop, every CI log
  • Factor III: store config in the environment — not in code
  • Factor IV: treat backing services as attached resources via URL/credential
  • Corollary: rotating a credential must never require a code change

The 12-Factor definition of "config" is everything that varies between deploys (dev/staging/prod). Passwords are a special, high-value subset.

AWS SSM Parameter Store

  • Hierarchical key-value store: /cloudair/prod/db/host
  • String — plain text (URLs, feature flags, thread counts)
  • SecureString — AES-256 via KMS; WithDecryption=True on read
  • StringList — comma-delimited set (subnet IDs, CORS origins)
  • Versioned automatically; rollback by version ARN
  • IAM + KMS policies control read/write per path prefix

Parameter Store's free tier covers most app-config use cases. Use Advanced tier (paid) for parameters >4 KB or TTL-based expiry enforcement.

Writing & Reading Parameters

# Write — CLI
aws ssm put-parameter \
  --name "/cloudair/$USER_ID/db/host" \
  --value "aurora-cluster.us-east-1.rds.amazonaws.com" \
  --type String --overwrite

# Read — boto3 (plain)
import boto3, os
ssm = boto3.client("ssm", region_name="us-east-1")
param = ssm.get_parameter(Name=f"/cloudair/{os.environ['USER_ID']}/db/host")
db_host = param["Parameter"]["Value"]

# Read — SecureString with decryption
secret = ssm.get_parameter(
    Name=f"/cloudair/{os.environ['USER_ID']}/db/password",
    WithDecryption=True)["Parameter"]["Value"]

Prefer get_parameters_by_path to bulk-load an entire config namespace at startup, minimising round trips.

AWS Secrets Manager

  • Purpose-built for secrets (passwords, API keys, OAuth tokens, TLS certs)
  • Stores JSON blobs; built-in templates for RDS, Redshift, DocumentDB
  • Automatic rotation via a Lambda function that calls the service API and updates the secret
  • Cross-account access, resource-based policies, audit via CloudTrail
  • Caching SDK (aws-secretsmanager-caching) avoids per-request API calls

Secrets Manager costs ~$0.40/secret/month + $0.05 per 10,000 API calls. For high-call-rate apps, always use the caching client.

Retrieving a Secret in Python

import boto3, json

sm = boto3.client("secretsmanager", region_name="us-east-1")

def get_db_creds(secret_name: str) -> dict:
    response = sm.get_secret_value(SecretId=secret_name)
    # get_secret_value returns either SecretString or SecretBinary
    raw = response.get("SecretString") or \
          response["SecretBinary"].decode("utf-8")
    return json.loads(raw)

creds = get_db_creds(f"cloudair/{os.environ['USER_ID']}/aurora")
conn = psycopg2.connect(
    host=creds["host"], dbname=creds["dbname"],
    user=creds["username"], password=creds["password"])

Automatic Secret Rotation

  • Attach a rotation Lambda (console wizard or custom) to a secret
  • Rotation lifecycle: createSecret → setSecret → testSecret → finishSecret
  • For RDS: Secrets Manager orchestrates a double-write (old + new valid simultaneously) before invalidating the old
  • Set RotationRules.AutomaticallyAfterDays or trigger on-demand
  • Apps reading via caching SDK automatically pick up the new value after TTL
aws secretsmanager rotate-secret \
  --secret-id "cloudair/$USER_ID/aurora" \
  --rotation-rules AutomaticallyAfterDays=30

Parameter Store vs. Secrets Manager

  • Use Parameter Store for non-secret config: feature flags, ARNs, URLs, capacity settings — free tier, hierarchy, deep IAM path control
  • Use Secrets Manager for anything you'd call a credential: DB passwords, API tokens, TLS keys — rotation, cross-account, audit
  • Both integrate with ECS task definitions, Lambda environment injection, and SAM AWS::SSM::Parameter references
  • Both are readable by Lambda without VPC; no NAT gateway required

Hybrid pattern: store the Secrets Manager ARN in Parameter Store so code only needs SSM read permission at startup; swap the ARN to rotate which secret is live without code changes.

Injecting Config into Lambda & Beanstalk

# SAM template — resolve SSM at deploy time
Environment:
  Variables:
    DB_HOST: !Sub "{{resolve:ssm:/cloudair/${UserID}/db/host}}"
    LOG_LEVEL: !Sub "{{resolve:ssm:/cloudair/${UserID}/log_level}}"

# Elastic Beanstalk — .ebextensions/config.config
option_settings:
  aws:elasticbeanstalk:application:environment:
    DB_HOST: "{{resolve:ssm:/cloudair/prod/db/host}}"
    SECRET_ARN: "arn:aws:secretsmanager:us-east-1:123456789012:\
secret:cloudair/prod/aurora-AbCdEf"

For Lambda, injecting at deploy time bakes the value at package time. For secrets that rotate, fetch at runtime inside the handler using the caching SDK instead.

Cloud Air Config & Secrets Architecture

  • /cloudair/$USER_ID/db/host — SSM String — Aurora cluster endpoint
  • /cloudair/$USER_ID/db/name — SSM String — database name
  • /cloudair/$USER_ID/feature/loyalty_enabled — SSM String — feature flag
  • cloudair/$USER_ID/aurora — Secrets Manager JSON — username + password (rotates every 30 days)
  • cloudair/$USER_ID/stripe_api_key — Secrets Manager String — payment key
