Module 15

Course Wrap-up

What Cloud Air became, the principles that got it there, and where to go next.

What Cloud Air Became

Before
  • Monolithic Python/Flask app on bare-metal
  • Single PostgreSQL instance
  • Manual deployments, nights and weekends
  • No audit trail, no distributed tracing
After
  • Serverless microservices on Lambda + API Gateway
  • DynamoDB + Aurora Serverless + S3 (polyglot)
  • SAM + CodePipeline CI/CD — deploy in minutes
  • X-Ray service map, structured logs, event audit trail

Every lab advanced one vertical slice of this refactor — not theory, a real working system.

The 6 Rs — Revisited

  • Rehost — lift-and-shift; used in Lab 1 to get the monolith running in EC2/Beanstalk
  • Replatform — Beanstalk → managed RDS, S3 static assets; minimal code change
  • Repurchase — swap self-managed auth for Cognito (Lab 7a)
  • Refactor / Re-architect — the dominant R in this course: microservices, Lambda, DynamoDB, Step Functions, EventBridge
  • Retire — modules decommissioned as the Strangler Fig replaced them (Lab 4b)
  • Retain — legacy billing service kept on-prem; connected via VPN + PrivateLink

Architecture Patterns Applied

  • Strangler Fig — incremental decomposition; traffic proxy routes new paths to Lambda, legacy paths to monolith
  • Saga / Orchestration — Step Functions coordinates the booking flow with compensating transactions
  • Event Sourcing + CQRS — DynamoDB Streams drives read-model projections; queries never hit the write store
  • Fan-out — SNS → SQS queues decouple notification, audit, and analytics consumers
  • Circuit Breaker + DLQ — failures are isolated; poison pills captured, never lost

The Twelve-Factor App — Where They Showed Up

  • III Config — SSM Parameter Store + Secrets Manager (Lab 3)
  • IV Backing services — DynamoDB, SQS, SNS treated as attached resources via env vars
  • VI Processes — stateless Lambda functions; all state externalised to DynamoDB or S3
  • IX Disposability — Lambda's cold start = fast startup; graceful shutdown via Lambda Extensions
  • XI Logs — structured JSON to stdout; CloudWatch aggregates, Logs Insights queries

The Twelve Factors are not a checklist to complete once — they are a lens to apply when designing any new service.

Habits to Carry Forward

  • Least privilege every time — policy review is the cheapest security control you have
  • Design access patterns first, schema second — especially true for DynamoDB
  • Everything in IaC — SAM/CloudFormation first; the console is for reading
  • Log structured JSON with correlation IDs from day one — retrofitting observability is expensive
  • Idempotency is not optional in at-least-once delivery systems
  • Small, frequent deploys — the fastest path to confidence and stability
