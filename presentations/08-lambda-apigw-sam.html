Microservices with Lambda, API Gateway & SAM

Deep-dive: Lambda internals, API Gateway integration patterns, and SAM as the Infrastructure-as-Code tool that ties them together for Cloud Air.

Lambda Deep Dive — Handler Contract

import json, boto3, os

dynamodb = boto3.resource("dynamodb")  # initialised once, reused across warm invocations
table = dynamodb.Table(os.environ["FLIGHTS_TABLE"])

def handler(event, context):
    origin = event["pathParameters"]["origin"]
    dest   = event["pathParameters"]["dest"]
    items  = table.query(
        KeyConditionExpression="origin = :o AND dest = :d",
        ExpressionAttributeValues={":o": origin, ":d": dest}
    )["Items"]
    return {"statusCode": 200,
            "body": json.dumps(items),
            "headers": {"Content-Type": "application/json"}}

event is the raw event JSON parsed to a dict. context exposes remaining_time_in_millis(), function name, request ID, and the invoked ARN.

Lambda Execution Environment

  • Init phase: download deployment package, start runtime, execute module-level code
  • Invoke phase: call handler function; execution environment may be reused ("warm")
  • Shutdown: environment frozen after configurable idle timeout (~5–15 min)
  • Anything initialised outside the handler survives across warm invocations — SDK clients, DB connections, loaded config
  • Concurrency = number of simultaneous execution environments; default burst limit 3,000/region
  • Reserved concurrency caps a function; Provisioned concurrency pre-warms environments

API Gateway — Integration Types

  • Lambda Proxy integration — full HTTP request forwarded as-is; handler owns the response shape (status, headers, body). Simplest, recommended default.
  • Lambda Custom integration — mapping templates transform request/response using VTL; rarely needed for new builds
  • HTTP / HTTP_PROXY — forward to any HTTP endpoint (the monolith!); no Lambda needed
  • Mock — return a static response; useful for CORS pre-flight and contract stubs
  • REST API (v1) vs. HTTP API (v2) — HTTP API is lower cost, lower latency, recommended unless you need usage plans or request validation

Cloud Air uses HTTP API (v2) for new microservices and a REST API (v1) for the legacy proxy route because REST API supports per-stage throttling and usage plans.

API Gateway — Request Flow

  • Client → DNS / CloudFront edge → API Gateway regional endpoint
  • API GW evaluates: authorizer (JWT / Lambda) → throttle limits → route matching
  • Matched route invokes the integration (Lambda, HTTP proxy, or mock)
  • Integration response mapped back to HTTP response
  • API GW adds X-Amzn-RequestId and X-Amzn-Trace-Id headers for tracing
  • Access logs and execution logs land in CloudWatch; enable $context.integrationLatency to surface Lambda cold starts

SAM — Serverless Application Model

  • CloudFormation extension: transforms AWS::Serverless::* resources into standard CFn
  • Opinionated shorthand for the Lambda + API GW + IAM role triad
  • sam build — resolves dependencies, packages artifacts into .aws-sam/
  • sam deploy --guided — uploads to S3, executes CloudFormation changeset
  • sam local invoke / sam local start-api — runs Lambda in Docker locally
  • Integrated with sam sync for fast incremental deploys during development

SAM is not a new deploy tool — it is a CloudFormation transform. Every sam deploy produces a standard CloudFormation stack you can inspect, rollback, and manage with existing CFn tooling.

SAM Template — Flight Search Service

AWSTemplateFormatVersion: '2010-09-09'
Transform: AWS::Serverless-2016-10-31

Parameters:
  UserID: { Type: String }

Globals:
  Function:
    Runtime: python3.12
    Environment:
      Variables:
        FLIGHTS_TABLE: !Sub "cloudair-${UserID}-flights"

Resources:
  FlightSearchFunction:
    Type: AWS::Serverless::Function
    Properties:
      Handler: search.handler
      CodeUri: src/
      Policies: [AmazonDynamoDBReadOnlyAccess]
      Events:
        SearchApi:
          Type: HttpApi
          Properties:
            Path: /flights/{origin}/{dest}
            Method: GET

SAM Build & Deploy Workflow

# First deploy — interactive guided setup
sam build
sam deploy --guided \
  --stack-name cloudair-$USER_ID-search \
  --parameter-overrides UserID=$USER_ID \
  --capabilities CAPABILITY_IAM \
  --region us-east-1

# Subsequent deploys — use samconfig.toml generated by --guided
sam build && sam deploy

# Local testing before deploy
sam local invoke FlightSearchFunction \
  --event events/search-event.json

sam build respects .gitignore and only packages source — never credentials. The samconfig.toml file commits safely to source control.

Lambda Layers & Dependency Management

  • Layers — shared ZIP of libraries, mounted at /opt in the execution environment
  • Use for: boto3 upgrades, shared utility code, large ML models, compiled binaries
  • Up to 5 layers per function; total unzipped size ≤ 250 MB
  • SAM: AWS::Serverless::LayerVersion with ContentUri pointing to a requirements folder
  • Alternative: Lambda container images — package everything in a Docker image, no size limit on the image itself (up to 10 GB ECR image)
# SAM — reference a layer in a function
Layers:
  - !Ref CloudAirUtilsLayer
  - arn:aws:lambda:us-east-1:123456789012:layer:aws-sdk-pandas:8

Testing Lambda Functions

  • Unit tests — mock boto3 with moto; call handler(event, {}) directly
  • Local integration — sam local start-api; call via curl or Postman against localhost:3000
  • Cloud integration — deploy to a personal stack (cloudair-$USER_ID-dev); test against real AWS services
  • Contract testing — validate event shapes with JSON Schema; catch API Gateway ↔ Lambda mismatches early
# moto-based unit test pattern
from moto import mock_dynamodb
import boto3, json
from search import handler

@mock_dynamodb
def test_search_returns_flights():
    # set up mock table, seed data, then:
    resp = handler({"pathParameters": {"origin":"JFK","dest":"LAX"}}, {})
    assert resp["statusCode"] == 200
