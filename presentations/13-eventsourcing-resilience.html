Event Sourcing, CQRS & Designing for Resilience

Formalizing Cloud Air's event-driven model and building systems that fail gracefully.

Event Sourcing

  • Instead of storing current state, store the ordered sequence of events that led to that state
  • Current state = fold (reduce) over the event log
  • Benefits: full audit trail, time-travel debugging, event replay to build new projections
  • Cloud Air bookings: SeatHeld → PaymentProcessed → BookingConfirmed → BoardingPassIssued — each event is immutable, append-only
# Append an event to DynamoDB event store
ddb = boto3.resource("dynamodb")
table = ddb.Table(f"cloudair-{USER_ID}-events")
table.put_item(Item={
    "pk": f"BOOKING#{booking_id}",
    "sk": f"EVENT#{timestamp}#{event_type}",
    "eventType": event_type,
    "payload": payload,
    "version": version
})

CQRS — Command Query Responsibility Segregation

  • Split the model into two paths: Commands (write, modify state) and Queries (read, never modify)
  • Commands write to the event store; projections (read models) are built asynchronously from those events
  • Enables independent scaling of read and write paths — Cloud Air's flight-search read replica scales to handle millions of queries without contending with booking writes
  • AWS implementation: DynamoDB Streams (events) → Lambda (projector) → ElasticSearch / DynamoDB (read model)

CQRS pairs naturally with event sourcing but is independently useful — you can apply CQRS without full event sourcing.

DynamoDB Streams as an Event Feed

# Lambda handler receiving DynamoDB Stream records
def handler(event, context):
    for record in event["Records"]:
        if record["eventName"] != "INSERT":
            continue
        new_image = record["dynamodb"]["NewImage"]
        event_type = new_image["eventType"]["S"]
        payload    = json.loads(new_image["payload"]["S"])

        if event_type == "BookingConfirmed":
            # Project into the read model
            update_booking_summary(payload)
            # Fan out to EventBridge
            publish_domain_event(event_type, payload)

DynamoDB Streams gives ordered, at-least-once delivery per partition key — treat the projector as idempotent.

Idempotency

  • At-least-once delivery means your handler will receive duplicates — design for it
  • Pattern: store a deduplication key (message ID + event type) in DynamoDB with a TTL; check before processing
  • Lambda Powertools idempotent decorator automates this pattern with DynamoDB persistence
from aws_lambda_powertools.utilities.idempotency import (
    idempotent, DynamoDBPersistenceLayer)

persistence = DynamoDBPersistenceLayer(
    table_name=f"cloudair-{USER_ID}-idempotency")

@idempotent(persistence_store=persistence)
def handler(event, context):
    # Safe to invoke multiple times — second call returns cached result
    return process_booking(event["bookingId"])

Retry Strategies & Exponential Backoff

  • Naive fixed-interval retries amplify load during outages — use exponential backoff with jitter
  • boto3 uses adaptive retry by default; tune with Config(retries={"max_attempts": 5, "mode": "adaptive"})
  • SQS visibility timeout acts as the retry interval for queue consumers — set it longer than your processing P99
  • Step Functions Retry block supports BackoffRate and MaxAttempts natively
from botocore.config import Config

ddb = boto3.client("dynamodb", config=Config(
    retries={"max_attempts": 5, "mode": "adaptive"}
))

Dead-Letter Queues & Poison Pills

  • A poison-pill message — one that always causes a processing error — will reappear until it hits maxReceiveCount, then moves to the DLQ
  • DLQ should trigger an alert (SNS → email/Slack) and a separate Lambda for inspection/requeue
  • EventBridge rules also support a DLQ for failed target invocations
  • Lambda async invocations have a dedicated DLQ configuration (FunctionEventInvokeConfig)
aws sqs set-queue-attributes \
  --queue-url $QUEUE_URL \
  --attributes '{"RedrivePolicy": "{\"deadLetterTargetArn\":\"'$DLQ_ARN'\",\"maxReceiveCount\":\"3\"}"}'

Designing for Failure — Key Patterns

  • Circuit Breaker — stop calling a failing downstream; fast-fail until it recovers
  • Bulkhead — isolate thread/connection pools so one service's failure doesn't starve others
  • Timeout everywhere — Lambda has a max 15 min timeout; always set it lower than your caller's timeout
  • Graceful degradation — Cloud Air serves cached flight data when the live inventory service is unhealthy
  • Health checks — ALB target groups, Route 53 health checks, and Lambda ProvisionedConcurrency warmup

AWS Lambda Powertools' BatchProcessor handles partial batch failures — report individual item failures so SQS only requeues failed records, not the whole batch.

EventBridge — Archive & Replay for Resilience

# Create an archive for all booking events
aws events create-archive \
  --archive-name cloudair-$USER_ID-booking-archive \
  --event-source-arn arn:aws:events:us-east-1:$ACCT:event-bus/cloudair-$USER_ID-bus \
  --event-pattern '{"source":["cloudair.booking"]}' \
  --retention-days 90

# Replay events to a target bus (e.g., test a new consumer)
aws events start-replay \
  --replay-name test-new-consumer \
  --event-source-arn arn:aws:events:us-east-1:$ACCT:archive/cloudair-$USER_ID-booking-archive \
  --event-start-time 2025-01-01T00:00:00Z \
  --event-end-time   2025-01-02T00:00:00Z \
  --destination '{"Arn":"arn:aws:events:us-east-1:$ACCT:event-bus/cloudair-$USER_ID-bus"}'

Replay enables testing new consumers against production-shaped historical data — a critical resilience and correctness practice.
