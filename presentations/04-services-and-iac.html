AWS Services, Interfacing & Infrastructure as Code

Service overview, CLI/SDK/API access patterns, authentication, and deploying the Cloud Air baseline with CloudFormation and Elastic Beanstalk.

AWS Service Landscape for Cloud Air

  • Compute: EC2, Lambda, Elastic Beanstalk, ECS/Fargate
  • Networking: VPC, ALB, API Gateway, Route 53, CloudFront
  • Storage & Data: S3, DynamoDB, RDS Aurora, ElastiCache
  • Messaging: SQS, SNS, EventBridge, Kinesis
  • Developer Tools: CodeCommit, CodeBuild, CodeDeploy, CodePipeline, Cloud9
  • Operations: CloudWatch, X-Ray, CloudTrail, AWS Config, Systems Manager
  • IaC: CloudFormation, CDK, SAM

This module focuses on the interfacing layer and the IaC tooling. Service-specific deep dives happen in later modules.

Three Ways to Interface with AWS

  • Console — rapid exploration and one-off operations. Never the source of truth for production infrastructure.
  • AWS CLI — scriptable; every service operation is a subcommand. Integrates into CI/CD pipelines and shell scripts.
  • SDK (boto3) — full programmatic control; paginators, waiters, event system. What your application code uses.
# CLI: describe a stack
aws cloudformation describe-stacks \
  --stack-name cloudair-${USER_ID}-base \
  --query "Stacks[0].StackStatus"

# SDK equivalent
import boto3, os
cf = boto3.client("cloudformation")
status = cf.describe_stacks(
    StackName=f"cloudair-{os.environ['USER_ID']}-base"
)["Stacks"][0]["StackStatus"]

Authentication & the Credential Provider Chain

  1. Explicit credentials passed to boto3.client() — never do this in application code
  2. AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY env vars
  3. Named profile in ~/.aws/credentials (selected via AWS_PROFILE)
  4. Container/EC2/Lambda execution role — what Cloud9 and Lambda use ← always prefer this

Instance roles deliver time-limited STS tokens that rotate automatically. This is how the Cloud Air application authenticates to every AWS service: no long-term credentials stored anywhere.

Infrastructure as Code — Why It Matters

Repeatability
The same template deployed 50 times produces identical infrastructure — critical for multi-region disaster recovery
Auditability
Every infrastructure change is a Git commit; who changed what and when is always traceable
Drift detection
CloudFormation tracks declared vs actual state; DRIFTED resources are flagged automatically
Cost & compliance
Templates are the enforcement point for tagging policies, encryption, and network isolation standards

Clicking through the console to build Cloud Air's baseline is not acceptable in a production context. Lab 2a makes the template the authoritative record.

CloudFormation — Template Anatomy

AWSTemplateFormatVersion: "2010-09-09"
Description: Cloud Air baseline stack

Parameters:
  UserId:
    Type: String

Resources:
  FlightSearchBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub "cloudair-${UserId}-search-assets"
      Tags:
        - Key: Owner
          Value: !Sub "cloudair-${UserId}"

  BookingTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: !Sub "cloudair-${UserId}-bookings"
      BillingMode: PAY_PER_REQUEST
      AttributeDefinitions:
        - AttributeName: PK
          AttributeType: S
      KeySchema:
        - AttributeName: PK
          KeyType: HASH

CloudFormation — Stack Operations

# Deploy (create or update) with a change set
aws cloudformation deploy \
  --template-file labs/files/lab2/cloudair-base.yaml \
  --stack-name cloudair-${USER_ID}-base \
  --parameter-overrides UserId=${USER_ID} \
  --capabilities CAPABILITY_NAMED_IAM \
  --tags Owner=cloudair-${USER_ID}

# Describe outputs (e.g. S3 bucket name, DynamoDB table ARN)
aws cloudformation describe-stacks \
  --stack-name cloudair-${USER_ID}-base \
  --query "Stacks[0].Outputs"

--capabilities CAPABILITY_NAMED_IAM is required when the template creates IAM resources with custom names. Never skip it blindly — verify what IAM resources the template actually creates.

Elastic Beanstalk — Deploying the Monolith

  • Elastic Beanstalk is a PaaS layer over EC2 + ALB + Auto Scaling — it manages the underlying infrastructure while letting you control the platform
  • Deployment model: zip your code, push to S3, Beanstalk handles rolling deploys, health checks, and rollback
  • Environment variables in Beanstalk replace hardcoded config — first step toward Factor III compliance
  • The Cloud Air monolith targets Python 3.11 on AL2023 — platform ARN in lab2b template
# Lab 2b: create the Beanstalk environment
aws elasticbeanstalk create-environment \
  --application-name cloudair-${USER_ID} \
  --environment-name cloudair-${USER_ID}-monolith \
  --solution-stack-name "64bit Amazon Linux 2023 v4 running Python 3.11" \
  --option-settings file://labs/files/lab2/eb-options.json

Beanstalk Environment Variables & Config Files

# .ebextensions/cloudair.config — Factor III: config from environment
option_settings:
  aws:elasticbeanstalk:application:environment:
    AWS_REGION: us-east-1
    DB_TABLE_NAME: cloudair-user1-bookings
    LOG_LEVEL: INFO
    SSM_PREFIX: /cloudair/user1

# In app code — no hardcoding, no config.py
import os
TABLE = os.environ["DB_TABLE_NAME"]
PREFIX = os.environ["SSM_PREFIX"]

These values will later move to SSM Parameter Store (Lab 3a). The Beanstalk option settings serve as a bridge: the code is already factor-compliant, the config location improves in Lab 3.

IaC Options at a Glance

  • CloudFormation — JSON/YAML; AWS-native; full feature coverage; change sets; drift detection. Used for the Cloud Air base stack.
  • AWS CDK — TypeScript/Python/Java constructs that compile to CloudFormation. Higher abstraction; best for complex, programmatically generated stacks.
  • AWS SAM — CloudFormation extension with shorthand for Lambda, API GW, Step Functions. Used from Lab 4a onwards for serverless services.
  • Terraform — multi-cloud HCL; not used in this course but the concepts transfer directly.

Rule of thumb: CloudFormation for shared infrastructure (VPC, DynamoDB, IAM); SAM for application-layer serverless resources. CDK when the infra logic is complex enough to warrant a programming language.
