Module 10

Polyglot Persistence & DynamoDB Best Practices

Right database per job, DynamoDB single-table design, access patterns first, and where DAX fits into the Cloud Air read path.

Polyglot Persistence

  • Microservices own their data — each service chooses the store that best fits its access patterns
  • Relational (Aurora) — strong consistency, complex queries, transactional writes (Cloud Air: booking ledger)
  • Key-value / document (DynamoDB) — single-digit millisecond reads at any scale (Cloud Air: flight inventory, loyalty points, sessions)
  • Search (OpenSearch) — full-text and faceted queries (Cloud Air: fare search with filters)
  • Cache (ElastiCache / DAX) — microsecond reads for hot data (Cloud Air: price cache)
  • Object store (S3) — boarding passes, receipts, ML training data

The anti-pattern is forcing every service to share a single Aurora instance because "that's what we know." Each service's query shape should drive the storage decision.

DynamoDB Fundamentals

  • Fully managed, serverless, multi-AZ replicated key-value + document store
  • Every item identified by a partition key (PK) and optional sort key (SK)
  • Data distributed across partitions by hashing PK; a single partition holds ≤10 GB
  • Reads: GetItem (single), Query (same PK, SK range), Scan (avoid in hot paths)
  • Writes: PutItem, UpdateItem, DeleteItem, TransactWriteItems (up to 100 items, atomic)
  • Capacity modes: on-demand (pay per RCU/WCU) or provisioned (reserve + auto-scaling)

1 RCU = one strongly consistent read of an item ≤ 4 KB. 1 WCU = one write of an item ≤ 1 KB. Model your capacity before choosing provisioned mode.

Access Patterns First

  • DynamoDB is not a flexible query engine — you must know your access patterns before schema design
  • List every query the application will execute: "get all flights from JFK on 2025-07-04", "get a booking by PNR", "list all bookings for user U"
  • Each access pattern must be answerable with a GetItem or Query — never a Scan
  • For Cloud Air Flight Search: 3 primary access patterns drive the entire key design
# Cloud Air access patterns
# AP1: Get flights by route+date
#   PK=ROUTE#JFK#LAX  SK=DATE#2025-07-04#FLIGHT#CA201
# AP2: Get a single flight by ID
#   PK=FLIGHT#CA201   SK=METADATA
# AP3: Get all departures from an airport on a date
#   GSI1PK=ORIGIN#JFK  GSI1SK=DATE#2025-07-04

Single-Table Design

  • Store all entity types in one table, distinguished by composite key patterns
  • Eliminates cross-table joins; all related data retrieved in a single Query
  • Use generic attribute names: PK, SK, GSI1PK, GSI1SK, EntityType
  • Overloaded keys: the same PK=USER#U001 anchor can have SK=PROFILE, SK=BOOKING#B001, SK=LOYALTY
  • Trade-off: flexibility of SQL is replaced by intentional, pre-modelled access patterns

Rick Houlihan's "NoSQL Design for DynamoDB" paper (AWS docs) is the canonical reference. Single-table is not always correct — if your access patterns are simple and few, a multi-table design can be clearer.

Global Secondary Indexes (GSIs)

  • A GSI projects a subset of attributes onto a new PK/SK pair; separate read capacity
  • Supports access patterns the base table cannot answer without a Scan
  • Up to 20 GSIs per table; each incurs storage and write amplification
  • GSI overloading: GSI1PK / GSI1SK carry different semantic values per entity type — one GSI, multiple access patterns
  • Consistency: GSIs are eventually consistent — propagation lag is typically <1 s but non-zero
aws dynamodb create-table \
  --table-name cloudair-$USER_ID-main \
  --attribute-definitions AttributeName=PK,AttributeType=S \
    AttributeName=SK,AttributeType=S \
    AttributeName=GSI1PK,AttributeType=S \
  --key-schema AttributeName=PK,KeyType=HASH \
    AttributeName=SK,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --global-secondary-indexes file://gsi.json

Capacity Modes & Hot Partitions

  • On-demand: scales instantly to any throughput; ideal for spiky, unpredictable loads (Cloud Air: fare search during a sale event)
  • Provisioned + auto-scaling: set min/max RCU/WCU; auto-scaling reacts to CloudWatch metrics; lower cost at steady, predictable load
  • Hot partition anti-pattern: monotonically increasing PK (e.g. timestamp) funnels all writes to one partition; use a shard suffix or randomise the PK prefix
  • Write sharding example: PK = "BOOKING#" + str(random.randint(0, 9)) spreads writes; read by querying all 10 shard keys and merging

A single DynamoDB partition is capped at 1,000 WCU and 3,000 RCU. Designing around this limit is mandatory for high-throughput write workloads.

DynamoDB Accelerator (DAX)

  • In-memory write-through cache, DynamoDB-compatible API — microsecond read latency
  • Drop-in replacement: change boto3.resource("dynamodb") to a DAX client endpoint
  • Ideal for: high-read workloads, hot items, repeated identical reads (fare display)
  • Deployed as a cluster inside your VPC; Lambda must be VPC-attached to reach it
  • Not appropriate for: strongly consistent reads (DAX cannot serve them), write-heavy tables
import amazondax, boto3
client = amazondax.AmazonDaxClient.resource(
    endpoint_url="dax://cloudair-dax.xyz.dax.us-east-1.amazonaws.com",
    region_name="us-east-1")
table = client.Table("cloudair-prod-main")
