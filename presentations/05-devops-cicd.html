Module 5

Gaining Agility: DevOps & CI/CD

Culture, practices, and the AWS toolchain that makes Cloud Air deployments safe, fast, and repeatable.

DevOps Is Not a Team Name

  • DevOps is a culture and set of practices that unite development and operations around shared ownership of the delivery lifecycle
  • Core pillars: flow (small batches, short cycles), feedback (fast detection of issues), continual learning (blameless postmortems, improvement loops)
  • Cloud Air's pre-migration process: quarterly releases, siloed ops team, no automated tests — the opposite of DevOps
  • Target: multiple deployments per day; every PR is a potential production release

AWS CI/CD services are the technical enablers. The culture change is harder and must come first — the pipeline is the contractual expression of the team's commitments.

CI/CD Pipeline Stages

  • Source — code commit triggers the pipeline; artefact stored in S3; version immutable
  • Build — compile, package, run unit tests, produce deployment artefact; no manual steps
  • Test — integration tests, security scanning (SAST), dependency vulnerability check
  • Deploy (staging) — automated deploy to a production-parity environment; smoke tests run automatically
  • Approval — optional manual gate (only for production in regulated contexts)
  • Deploy (production) — rolling or blue/green deploy; automated rollback on CloudWatch alarm breach

The pipeline is code (stored in the same repo as the application). Changing the pipeline requires the same review process as changing the application.

AWS CodeCommit

  • Fully managed Git hosting in AWS — IAM-authenticated, no SSH key management needed
  • Triggers on push events feed directly into CodePipeline source stages
  • Pull request approval rules integrate with IAM — enforce review gates without third-party tools
  • For Cloud Air: one repo per service post-decomposition; one repo for shared IaC templates
# Configure CodeCommit credential helper — uses AWS STS tokens
git config --global credential.helper \
  '!aws codecommit credential-helper $@'
git config --global credential.UseHttpPath true

# Clone the Cloud Air monolith repo
git clone https://git-codecommit.us-east-1.amazonaws.com/v1/repos/\
cloudair-${USER_ID}-monolith

AWS CodeBuild

  • Managed build service — no Jenkins servers to patch, no build agents to scale
  • Build spec in buildspec.yml at the repo root — version-controlled alongside the code
  • Environment: Docker image (AWS standard or custom ECR image); ephemeral per build
  • IAM role scoped to the minimum permissions the build needs — not admin
version: 0.2
phases:
  install:
    runtime-versions:
      python: 3.11
    commands:
      - pip install -r requirements.txt
  build:
    commands:
      - python -m pytest tests/unit -v
      - pip install --target dist/ -r requirements.txt
      - cp -r src/ dist/
artifacts:
  files: ["dist/**/*", "appspec.yml"]
  discard-paths: no

AWS CodeDeploy

  • Automates application deployments to EC2, Beanstalk, Lambda, and ECS
  • Deployment strategies: In-place (rolling), Blue/Green (swap traffic after validation), Canary/Linear (shift Lambda traffic incrementally)
  • appspec.yml defines lifecycle hooks: BeforeInstall, AfterInstall, ValidateService
  • Automatic rollback on CloudWatch alarm or hook failure — no human needed at 2 AM
version: 0.0
os: linux
files:
  - source: /dist
    destination: /var/app/current
hooks:
  AfterInstall:
    - location: scripts/restart_app.sh
      timeout: 60
  ValidateService:
    - location: scripts/health_check.sh
      timeout: 30

AWS CodePipeline

  • Orchestrates the full CI/CD workflow — connects Source → Build → Test → Deploy as a directed graph
  • Each stage produces and consumes artefacts via S3; stages run sequentially; parallel actions within a stage are supported
  • Native integrations: CodeCommit, CodeBuild, CodeDeploy, Elastic Beanstalk, CloudFormation, Lambda, ECS, manual approval actions
  • Pipeline definition is itself a CloudFormation resource — IaC for your IaC pipeline
# Trigger a pipeline execution manually (useful after infra changes)
aws codepipeline start-pipeline-execution \
  --name cloudair-${USER_ID}-pipeline

# View the latest execution state
aws codepipeline get-pipeline-state \
  --name cloudair-${USER_ID}-pipeline \
  --query "stageStates[*].{Stage:stageName,Status:latestExecution.status}"

Cloud Air Pipeline Architecture

Source stage
CodeCommit push to main triggers pipeline; artefact version locked in S3
Build stage
CodeBuild: pytest unit tests → package to zip → SAM build for Lambda services
Deploy-staging stage
CloudFormation change set execute; Beanstalk rolling deploy; smoke test Lambda invocation
Deploy-prod stage
Blue/green on Beanstalk for monolith; CodeDeploy linear-10-percent canary for Lambda functions

In practice, the pipeline itself is a CloudFormation stack (a pipeline.yaml template is illustrative of this pattern) — the pipeline manages itself after the first aws cloudformation deploy.

Deployment Strategies Compared

  • All-at-once: fastest; full downtime risk. Acceptable only for non-production environments.
  • Rolling: replaces instances in batches; reduced capacity during deploy; no extra cost. Good default for Beanstalk.
  • Blue/Green: fully parallel environments; instant cutover; easy rollback; doubles cost during deploy window. Required for zero-downtime releases.
  • Canary / Linear (Lambda): traffic shifts by percentage over time; CloudWatch alarms auto-rollback on elevated error rate. Best for serverless where "instance" cost is irrelevant.

Cloud Air commitment: the booking service never has >30 s of downtime during a deploy. Blue/Green on the ALB is the only strategy that guarantees this.
