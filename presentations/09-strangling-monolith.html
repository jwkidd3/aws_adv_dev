Module 9

Strangling the Monolith

Incremental decomposition of the Cloud Air legacy app using the Strangler Fig pattern — zero big-bang rewrites, continuous delivery throughout.

The Strangler Fig Pattern

  • Named after the tropical fig that grows around a host tree, eventually replacing it
  • Core idea: route new functionality and extracted endpoints to the new system; proxy everything else to the old system
  • The monolith shrinks incrementally; the new system grows; at some point the monolith is empty and is decommissioned
  • Business continues operating throughout — no freeze, no flag day
  • Each extraction is independently reversible (re-route back to the monolith if the new service fails)

Martin Fowler coined the pattern in 2004, drawing the analogy from strangler figs in Queensland. The botanical metaphor holds: the host tree (monolith) provides support while the new organism (services) matures.

The Facade / Proxy Layer

  • A single entry point intercepts all requests before they reach the monolith
  • For Cloud Air: API Gateway acts as the facade
  • Routes that have been extracted → Lambda integrations
  • Routes not yet extracted → HTTP_PROXY integration to the Beanstalk monolith URL
  • Clients see one hostname; internal routing is invisible to them
  • Over time: add more Lambda routes, shrink the proxy routes
# SAM — proxy catch-all to legacy monolith
LegacyProxy:
  Type: HttpApi
  Properties:
    RouteKey: "ANY /{proxy+}"
    Integration:
      Type: HTTP_PROXY
      Uri: !Sub "http://${MonolithUrl}/{proxy}"
      Method: ANY

Anti-Corruption Layer (ACL)

  • The monolith's domain model is often tangled — IDs as strings, denormalised data, legacy naming
  • An Anti-Corruption Layer (from DDD) translates between the legacy model and the new service's clean domain model
  • Implemented as a thin adapter in the Lambda function or as a dedicated translation Lambda
  • Prevents the new service from inheriting the monolith's design flaws
  • When the monolith is finally decommissioned, the ACL is removed — not refactored

Example: Cloud Air monolith stores flight_id as a 12-char alphanumeric legacy PNR. The new Flight Search service uses a structured UUID. The ACL translates at the boundary.

Extraction Sequence — Cloud Air

  1. Flight Search (Lab 4a) — highest read volume, no writes, easiest to isolate
  2. Loyalty Points read — read-only projection, decouple DB first
  3. Booking Create — requires Step Functions orchestration (Lab 5b)
  4. Payments — extract after secrets management is in place (already done — Lab 3)
  5. Notifications — async; fan-out via SNS (Lab 6)
  6. Admin / reporting — last; lowest business priority, most internal coupling

The ordering is not arbitrary — it follows the dependency graph and the risk profile. Extract leaf nodes first; extract nodes with many dependencies last.

Traffic Shifting Strategies

  • Hard cutover — update the API GW route; instant 100% switch. Simple but highest risk.
  • Lambda weighted aliases — split traffic between two Lambda function versions (prod:BLUE vs prod:GREEN); shift weight gradually
  • Feature flags via SSM — handler reads a flag; routes internally to old or new implementation path
  • Shadow mode — proxy still serves the response, but new Lambda is invoked async for comparison logging
aws lambda update-alias \
  --function-name cloudair-$USER_ID-search \
  --name prod \
  --routing-config AdditionalVersionWeights={"2"=0.10}

Data Separation — Breaking the Shared Database

  • The hardest part of strangling: shared schema means a single change can break multiple services
  • Step 1: identify which tables a service owns vs. which it reads from others
  • Step 2: create a read replica / projection in the new service's store (DynamoDB)
  • Step 3: migrate writes to go through the new service's API; retire direct DB calls from the monolith
  • Step 4: remove the shared table access from the monolith IAM policy

Cloud Air's loyalty table is the first to split: the new Loyalty Lambda writes to its own DynamoDB table; the monolith reads via the Loyalty API rather than the shared DB.

Observability During Migration

  • Deploy X-Ray tracing on both the monolith (Beanstalk) and the Lambda functions
  • Ensure trace IDs propagate through API Gateway → Lambda → downstream services
  • Compare p50 / p99 latency before and after extraction to detect regression
  • CloudWatch alarms on Lambda error rate; auto-rollback via CodeDeploy Lambda deployment
  • Structured logs: add route, version, user_id to every log line

You cannot safely strangulate a monolith without being able to compare the behaviour of old and new paths at runtime. Instrumentation is a pre-requisite, not an afterthought.

Cloud Air — Current Extraction State

  • After Lab 4b: Flight Search is served by Lambda; all other routes proxy to Beanstalk
  • API Gateway is the single entry point — DNS unchanged for all clients
  • Monolith still owns the Aurora DB; new Lambda has its own DynamoDB flights table
  • Next: Booking service extraction will require Step Functions for saga orchestration (Module 11)
  • The Beanstalk environment remains live; its route weight diminishes with each extraction

By the end of Day 3, the Cloud Air monolith handles only the admin routes. Everything customer-facing runs on Lambda. That is a complete Strangler Fig in three days.
