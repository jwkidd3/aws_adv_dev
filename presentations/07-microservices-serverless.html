Module 7

Microservices & Serverless

Why the Cloud Air monolith must change, how microservices solve the problem, and where serverless fits in the decomposition.

The Monolith Pain Points

  • Deployment coupling — a one-line bug fix in payments requires a full Cloud Air release
  • Scaling granularity — the whole app scales even when only search is under load
  • Technology lock-in — every team is forced onto the same Python/Flask/Postgres stack
  • Fault blast radius — an unhandled exception in loyalty brings down booking
  • Long feedback loops — integration tests take 45 min; deploys require a maintenance window

Cloud Air's ops team currently ships once per quarter because every release is a high-risk all-or-nothing event. Microservices change the unit of deployment.

Microservices — The Core Idea

  • Single responsibility: each service owns one bounded context (search, booking, loyalty, payments)
  • Independently deployable — separate pipeline, separate rollback
  • Independently scalable — search scales on read; payments scales on write
  • Communicate over well-defined APIs (REST, events, async messages)
  • Own their data store — no shared database between services
  • Small enough for one "two-pizza" team to understand end-to-end

Microservices Trade-offs

  • Gains: deploy velocity, fault isolation, polyglot stacks, elastic scaling per service
  • Costs: distributed-systems complexity — network latency, partial failures, eventual consistency
  • Observability debt: a request that crosses five services needs distributed tracing (X-Ray)
  • Data management: no cross-service JOINs; views must be projected or aggregated
  • Operational surface: more things to deploy, monitor, and secure

The correct answer to "should we microservice X?" is never obvious. Start with the seams where teams are already blocked on each other, not with an arbitrary split.

The Serverless Model

  • No servers to provision or patch — the cloud provider manages compute fleet
  • Pay per invocation/duration — zero cost at zero traffic
  • Scales to zero and back up automatically — no capacity planning
  • Ideal unit for microservices: one Lambda per domain operation or per API route
  • AWS serverless primitives: Lambda, API Gateway, DynamoDB on-demand, SQS/SNS/EventBridge, Step Functions, S3

Serverless does not mean "no operations" — it means operations responsibility shifts from infrastructure to code, configuration, and IAM policy.

Lambda Execution Model

  • Event-driven: invoked by a trigger — API Gateway, SQS, S3, EventBridge, schedules
  • Stateless handler: each invocation receives an event JSON and a context object
  • Execution environment lifecycle: Init → Invoke → (warm reuse | Shutdown)
  • Init (cold start): download + unzip package, run module-level code, initialise connections
  • Warm invocation: execution environment reused; connections persist
  • Max duration: 15 min. Memory: 128 MB – 10 GB. vCPU scales linearly with memory.

Cold starts are measured in hundreds of milliseconds for Python. Keep packages lean, initialise SDK clients outside the handler, and use Provisioned Concurrency only when p99 latency matters.

Cloud Air Target Architecture

  • Flight Search — API Gateway → Lambda → DynamoDB (read-heavy, cache-friendly)
  • Booking — API Gateway → Lambda → Step Functions → Aurora Serverless
  • Loyalty — EventBridge rule → Lambda → DynamoDB (async, points accrual)
  • Payments — API Gateway → Lambda → Secrets Manager → Stripe API
  • Notifications — SNS → Lambda → SES / push
  • Legacy monolith stays alive during migration, fronted by API Gateway as a pass-through proxy

This is the Strangler Fig in practice: new routes resolve to new Lambda functions; unextracted routes proxy to the monolith. The monolith shrinks over time without a big-bang rewrite.

Decomposition Strategy — Domain Boundaries

  • Use Domain-Driven Design bounded contexts to draw service lines
  • Choose seams that minimise cross-service synchronous calls
  • Data ownership test: if two services share a table, they are not independent
  • Start with the most independently deployable domain (Cloud Air: Flight Search)
  • Anti-corruption layer: adapter that translates old monolith schema to the new domain model

Conway's Law: your service boundaries will reflect your org chart. If your team owns search, make search a service. Don't let technical convenience override team autonomy.

Serverless Patterns for Cloud Air

  • Request-response — synchronous Lambda invocation via API Gateway (search, booking create)
  • Fan-out — SNS topic triggers multiple Lambda subscribers (post-booking: loyalty + notifications)
  • Queue worker — SQS + Lambda event-source mapping (retry-safe, at-least-once)
  • Orchestration — Step Functions state machine (multi-step booking saga with compensation)
  • Scheduled job — EventBridge Scheduler → Lambda (nightly loyalty recalculation)

Why Not Containers for Everything?

  • Containers (ECS/EKS) are the right choice when: long-running processes, stateful workloads, GPU, sub-100ms cold-start requirements, or need to package a complex runtime
  • Lambda is the right choice when: event-driven, short-lived, variable or bursty traffic, per-invocation cost model desired
  • Cloud Air uses both: the legacy monolith runs on Elastic Beanstalk (managed EC2); new microservices run on Lambda
  • The choice is not ideological — it is operational cost vs. control

Lambda container image support (up to 10 GB) narrows the gap further — you can package any runtime in a Docker image and deploy it as a Lambda function.
