Module 14

Security & Observability

Locking down Cloud Air with Cognito authorization and making every distributed call traceable with X-Ray.

Lambda Security Foundations

  • Execution role — least-privilege IAM role assumed by every Lambda invocation; no ambient credentials
  • Resource-based policy — controls who can invoke the function (API Gateway, EventBridge, SQS)
  • VPC placement — optionally attach Lambda to a VPC for private resource access; adds cold-start latency, requires NAT or VPC endpoints for AWS API calls
  • Environment variable encryption — KMS CMK encrypts env vars at rest; avoid plaintext secrets — use Secrets Manager instead
  • Code signing — enforce that only signed packages from trusted CodeArtifact sources can be deployed

Least privilege is not optional — over-permissioned Lambda functions are the single most common serverless security finding.

Amazon Cognito — User Identity for Cloud Air

  • User Pool — directory of users; handles sign-up, sign-in, MFA, password policies, forgot-password flows
  • Identity Pool — exchanges tokens for temporary AWS credentials; enables direct AWS service access from the client
  • OAuth 2.0 / OIDC flows: Authorization Code (web apps), Client Credentials (service-to-service), Implicit (legacy — avoid)
  • Hosted UI provides a drop-in sign-in page; custom domain + CNAME for branding
  • Triggers: pre-authentication Lambda, post-confirmation Lambda, pre-token-generation — customize the auth flow without forking Cognito

JWT Authorizer on API Gateway

# SAM template excerpt
CloudAirApi:
  Type: AWS::Serverless::Api
  Properties:
    StageName: prod
    Auth:
      DefaultAuthorizer: CognitoAuth
      Authorizers:
        CognitoAuth:
          JwtConfiguration:
            issuer: !Sub "https://cognito-idp.us-east-1.amazonaws.com/${UserPool}"
            audience: [!Ref UserPoolClient]
          IdentitySource: "$request.header.Authorization"
BookingFunction:
  Type: AWS::Serverless::Function
  Properties:
    Events:
      Api:
        Type: HttpApi
        Properties:
          ApiId: !Ref CloudAirApi
          Path: /bookings
          Method: POST

API Gateway validates the JWT signature and expiry before invoking Lambda — no token validation code in your handler.

Reading Claims in the Lambda Handler

def handler(event, context):
    # API Gateway HTTP API injects the validated JWT claims
    claims = event["requestContext"]["authorizer"]["jwt"]["claims"]
    user_id  = claims["sub"]            # Cognito user UUID
    email    = claims["email"]
    groups   = claims.get("cognito:groups", [])

    if "admins" not in groups:
        return {"statusCode": 403, "body": "Forbidden"}

    # Scope data access to the authenticated user
    bookings = get_bookings_for_user(user_id)
    return {"statusCode": 200, "body": json.dumps(bookings)}

Never trust a user-supplied ID in the request body — always extract identity from the validated JWT claims injected by API Gateway.

AWS X-Ray — Distributed Tracing

  • Instruments requests end-to-end: API Gateway → Lambda → DynamoDB → SNS → downstream Lambda
  • Trace — one end-to-end request; composed of segments (per service) and subsegments (per operation within a service)
  • Service Map — visual graph of all services and their error rates + latency — identify bottlenecks and failure paths at a glance
  • X-Ray SDK patches boto3 automatically — every AWS API call becomes a traced subsegment
  • Sampling rules control cost: default 5% + 1 req/s reservoir; tune per path/method

Enable active tracing on Lambda with a single SAM property: Tracing: Active — no code change required.

Instrumenting Lambda with the X-Ray SDK

from aws_xray_sdk.core import xray_recorder, patch_all

patch_all()  # patches boto3, requests, httplib

@xray_recorder.capture("process_booking")
def process_booking(booking_id: str):
    xray_recorder.current_subsegment().put_annotation("bookingId", booking_id)
    xray_recorder.current_subsegment().put_metadata("payload", {"id": booking_id})

    with xray_recorder.in_subsegment("fetch_inventory"):
        inventory = ddb.get_item(TableName=INVENTORY_TABLE,
                                  Key={"pk": {"S": booking_id}})
    return inventory

Annotations are indexed and searchable in the X-Ray console; metadata is stored but not indexed. Use annotations for correlation IDs and environment.

Lambda Powertools — Structured Logging + Tracing

from aws_lambda_powertools import Logger, Tracer

logger  = Logger(service="booking-service")
tracer  = Tracer(service="booking-service")

@tracer.capture_lambda_handler
@logger.inject_lambda_context(correlation_id_path="requestContext.requestId")
def handler(event, context):
    logger.info("Booking request received",
                extra={"bookingId": event["bookingId"],
                       "userId":    event["userId"]})
    result = process(event)
    logger.info("Booking complete", extra={"result": result})
    return result

Powertools emits JSON logs with cold_start, correlation_id, service, and xray_trace_id automatically — CloudWatch Logs Insights queries become trivial.

CloudWatch Logs Insights — Querying Distributed Traces

/* Find slow booking requests across all Cloud Air Lambdas */
fields @timestamp, service, correlation_id, bookingId, @duration
| filter service like /cloudair/
| filter @duration > 2000
| sort @timestamp desc
| limit 50
  • Use the shared correlation_id (injected by Powertools) to pivot from a Logs Insights result to the full X-Ray trace
  • CloudWatch ServiceLens stitches CloudWatch metrics, logs, and X-Ray traces into one view

Observability is not a feature to add at the end — instrument from the first Lambda and keep the correlation ID flowing through every downstream call.
